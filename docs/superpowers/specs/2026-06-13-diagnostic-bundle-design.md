# Diagnostic Bundle Design

## Goal

Add a practical diagnostics export so operators can capture enough system context to troubleshoot PTZ Command without manually copying logs, health panels, and version details into a support message like it is still 1998.

## Scope

This first Better Diagnostics slice adds:

- An **Export Diagnostics** action on the Diagnostics page.
- A redacted JSON diagnostic bundle generated by the server.
- A compact **Warnings / Errors** summary on the Diagnostics page.
- Client-side download handling for the bundle.

This pass is read-only. It does not run active network probes, reconnect devices, alter settings, create test users, or execute camera/mixer/switcher commands. Those can come later after the app has a clean export format and summary surface.

## Current State

The app already has:

- `/diagnostics` page showing device health, Hue bridge status, system load, and recent events.
- `/api/health/devices` for cameras, mixers, switchers, and displays.
- `/api/health/system` for CPU, memory, network throughput, process RSS, and uptime.
- `/api/logs` for persisted audit logs.
- `/api/logs/recent` for in-memory runtime logs.
- `/api/session-log` for the current session activity log.
- Centralized `logger` and persisted audit logging.

The missing operator workflow is a single exportable incident snapshot and a clearer top-level summary of warnings/errors.

## User Experience

On the Diagnostics page, add a secondary action beside Refresh:

- `Export Diagnostics`

Clicking it downloads a JSON file named with the app version and timestamp:

- `ptz-command-diagnostics-v1.7.8-2026-06-13T08-30-00.json`

The page also gains a compact summary section near the top:

- Warning/error count from recent runtime logs.
- Offline device count.
- Offline Hue bridge count.
- Last warning/error message, if one exists.

The existing device health, Hue bridge, system load, and recent events sections remain. This is not another page, because the app already has one and duplicating it would be a brave new form of clutter.

## Server API

Add:

- `GET /api/diagnostics/bundle`

The route returns JSON with:

```json
{
  "generatedAt": "2026-06-13T08:30:00.000Z",
  "version": "1.7.8",
  "runtime": {
    "nodeVersion": "v24.14.0",
    "platform": "darwin",
    "arch": "arm64",
    "pid": 12345,
    "uptimeSeconds": 1234,
    "workingDirectory": "/Users/..."
  },
  "system": {},
  "health": {},
  "recentLogs": [],
  "auditLogs": [],
  "sessionLog": [],
  "summary": {
    "offlineDevices": 0,
    "warnings": 0,
    "errors": 0,
    "lastProblem": null
  }
}
```

The route should reuse existing health/system/log/session data paths instead of inventing separate diagnostic math. Shared helper functions can be extracted from `server/routes/system.ts` if needed to avoid duplicating health assembly.

## Redaction

The diagnostic bundle must redact sensitive values before response:

- `password`
- `apiKey`
- `token`
- `secret`
- `accessToken`
- `refreshToken`
- `Authorization`
- URL usernames/passwords such as `rtsp://user:pass@camera/stream`

Redaction should be recursive and should preserve the object shape. Example:

```json
{
  "streamUrl": "rtsp://redacted:redacted@192.168.0.22/stream1",
  "apiKey": "redacted"
}
```

This export is meant to be shareable for troubleshooting. Shipping secrets inside the support bundle would be an impressively efficient way to make diagnostics worse than the bug.

## Client API

Add a diagnostics client method:

- `diagnosticsApi.exportBundle()`

It calls `/api/diagnostics/bundle`, downloads the response as a formatted JSON file, and surfaces failures as toasts on the Diagnostics page.

## Diagnostics Page Changes

Add:

- Export button beside Refresh.
- Export pending state.
- Summary cards for warnings, errors, and last problem.

The Warnings / Errors summary should derive from `/api/logs/recent`, using `level === "warn"` and `level === "error"`. It should not block rendering if logs fail to load; show neutral placeholders and keep the rest of Diagnostics usable.

## Access Control

The bundle is read-only but operationally sensitive. Require at least `operator` access for:

- `GET /api/diagnostics/bundle`

Existing Diagnostics page visibility stays unchanged unless current routing already restricts it elsewhere.

## Error Handling

If part of the bundle fails to load, the route should return a successful bundle with a `collectionErrors` array describing which section failed. A single failing log read should not make the whole diagnostic export useless, because partial evidence beats a pristine 500 response with no payload.

If the server cannot build any meaningful bundle, return a 500 with a concise message.

On the client, export failures show a toast:

- `Failed to export diagnostics`

## Testing

Add focused tests for pure diagnostics behavior:

- Recursive redaction masks sensitive keys.
- Recursive redaction masks URL credentials.
- Summary counts offline devices, warnings, and errors.
- Bundle assembly preserves partial data and records collection errors.

Run the existing project checks after implementation:

- `npm run test`
- `npm run check`
- `npm run build`

Browser smoke:

- Open the Diagnostics page.
- Confirm the Export Diagnostics button renders.
- Confirm the summary cards render without overlap.
- Trigger export when authenticated test access is available.

If no authenticated browser session is available, report that UI smoke was limited to signed-out route behavior and automated checks.

Update `README.md` and `CHANGELOG.md`.

## Out Of Scope

Not included in this slice:

- Active TCP/UDP device probing.
- FFmpeg preview probe execution.
- OBS/ATEM/X32 reconnect commands.
- Command trace recording.
- Persistent diagnostic bundle storage.
- Uploading diagnostics to any external service.
